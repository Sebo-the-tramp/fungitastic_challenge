\section{Introduction}

Fine-grained fungal recognition is hard for the same reasons many ecological benchmarks are hard: subtle inter-species differences, large intra-species variation, cluttered field backgrounds, and long-tailed class frequencies. FungiTastic is a useful benchmark for this setting because it extends earlier fungi datasets with more images, richer metadata, and segmentation masks for part of the corpus \cite{Picek_2022_WACV,picek2022automatic,picek2024fungitastic}.

The repo README now defines a broader paper than the earlier draft. Instead of asking only whether one DINOv3 token construction is best, the target manuscript studies three questions. First, how much does segmentation help, and does the answer change between no segmentation, oracle foreground masks, and SAM-3 predicted masks? Second, which frozen feature family transfers best to fungi: CLIP, SigLIP, MAE, DINOv1, DINOv2, DINOv3, or InternViT? Third, which lightweight classifier extracts the most from those features: prototype, $k$-NN, linear probe, or MLP?

This broader scope also changes what the paper should look like. The README asks for two teaser plots: one for a 200-class oracle-segmentation setting and one for a 2000-class SAM-3 setting. The body then expands those top-line plots with a backbone-family ablation and a classifier ablation. The LaTeX draft should therefore reflect that structure even before every plot has been exported.

Only part of the empirical grid is completed today. The repo already contains a concrete first slice: DINOv3 ViT-7B runs with MLP heads at 224 and 448 resolution, plus a 224 $k$-NN sweep. Those runs already suggest that mask-aware pooled patches are useful when fused with the global token, while patch-only descriptors are weak. However, they do not yet answer the full README-level questions about backbone ranking, 2000-class scaling, or classifier ordering across the entire feature-family set.

This draft is therefore written with two goals. It matches the final paper organization described in the README, and it preserves the completed DINOv3 evidence so the manuscript remains concrete and buildable while the remaining runs are added.
